PART A — BIG IDEAS FIRST (read this before the rest)
================================================================================

WHAT IS A DATA STRUCTURE?
-------------------------
A data structure is just a way of keeping and organizing information in the
computer so we can find it, add to it, change it, or delete it easily.

Think of it like different ways to store things in real life:
  - A numbered list on paper     -> like a VECTOR (linear)
  - A phone book sorted by name  -> like a MAP (non-linear / hash table)


LINEAR vs NON-LINEAR — WHAT DOES THAT MEAN?
-------------------------------------------

LINEAR data structure:
  Items are stored one after another, like beads on a string or people in
  a queue at the bank. To find something, you often walk through the list
  from start to end — one step at a time.

  Example in our project: VECTOR

NON-LINEAR data structure:
  Items are NOT stored in one simple straight line. The computer uses a
  smarter trick (like a hash / label system) to jump closer to the right
  item without checking every single thing.

  Example in our project: UNORDERED_MAP (hash map)


WHAT IS O(1), O(n) — EXPLAINED SIMPLY?
--------------------------------------

These describe HOW FAST an operation is when data grows.

O(1) — "Oh-one" — CONSTANT TIME (very fast, like magic lookup):
  Imagine a locker with your name on it. You walk straight to YOUR locker.
  You do not check every locker in the building.
  No matter if there are 10 lockers or 10,000, you still take ONE direct step.

  In code: map.find("RAB123A") — jump straight to that plate's record.

O(n) — "Oh-en" — LINEAR TIME (check one by one):
  Imagine a line of 100 lunch boxes with no labels. To find yours, you open
  box 1, then box 2, then box 3... until you find it. Worst case: check all n.

  In code: looping through every transaction in a vector to find one plate.

QUICK TABLE:
  O(1)  = 1 step (instant jump)       -> very good for lookups
  O(n)  = up to n steps (one by one)  -> fine when list is small

PART B — EVERY DATA STRUCTURE WE USE

--------------------------------------------------------------------------------
1. VECTOR  (LINEAR)
   Used as: transactionHistory_  and  tariffPolicies_
--------------------------------------------------------------------------------

WHAT IS A VECTOR? (explain like to a child)
  A vector is like a row of boxes glued together in a straight line.
  Box 1, Box 2, Box 3, Box 4...
  You always add a NEW box at the END of the row.
  You cannot easily insert in the middle without shifting everything.

  Real-life picture:
    [Receipt 1] [Receipt 2] [Receipt 3] [Receipt 4]  <- end, add here

TYPE: LINEAR (items in a straight sequence)

WHERE WE USE IT IN THE PROJECT:

  (A) vector<ParkingTransaction>  transactionHistory_
      -----------------------------------------------
      WHAT IT STORES:
        Every FINISHED parking session (after a car leaves and pays).
        Like a pile of old receipts in order: first exit, second exit, etc.

      WHY A VECTOR?
        - Cars only LEAVE once, so we only ADD a receipt at the end.
        - We do not need to search by plate super-fast in this list.
        - Adding at the end (push_back) is very fast.

      OPERATIONS:
        ADD at end     -> push_back(transaction)     -> O(1) amortized (fast)
        READ all       -> loop from 0 to end         -> O(n) (check each one)
        SEARCH by plate-> loop and compare plates    -> O(n) (one by one)

      USED IN:
        Task 4 — save completed parking when vehicle exits
        Task 5 — show all history, search one plate, daily revenue report

      SIMPLE EXAMPLE:
        Car RAB123A exits -> receipt goes in the next empty box at the end.
        Car RAC456B exits -> another box added after that.
        Revenue report -> walk through ALL boxes and add up the fees.


  (B) vector<TariffPolicy*>  tariffPolicies_
      -----------------------------------------
      WHAT IT STORES:
        Pointers to 3 small "price rule" objects at program start:
        Motorcycle price, Car price, Truck price.

      WHY A VECTOR?
        - Only 3 items, never changes during the program.
        - Simple list to loop once at startup to load default prices.

      TYPE: LINEAR (only 3 items in a tiny list)

      OPERATIONS:
        Loop 3 times at start -> fill currentPrices_ map


--------------------------------------------------------------------------------
2. UNORDERED_MAP  (NON-LINEAR — hash map)
   Used as: slots_  ,  activeVehicles_  ,  currentPrices_
--------------------------------------------------------------------------------

WHAT IS AN UNORDERED_MAP? (explain like to a child)
  Imagine a magical cupboard with labels on the doors.
  Each label is a UNIQUE name (like "C-B1" or "RAB123A").
  You say the name -> the cupboard opens THAT door directly.
  You do NOT open every door one by one.

  Real-life picture:
    Label "C-B1"     -> [ slot info inside ]
    Label "RAB123A"  -> [ parked car info inside ]
    Label "CAR"      -> [ price 1000 inside ]

TYPE: NON-LINEAR (uses hashing — not a simple straight line)

WHY O(1) FOR FIND / INSERT / UPDATE (on average):
  The computer uses the KEY (name) to calculate where the data lives.
  Like knowing locker #47 from your student ID — you go straight there.
  Average time: ONE step = O(1).

  Note: Finding a FREE parking slot by scanning all slots is still O(n)
  because we must LOOK at each slot until we find an empty matching one.
  But finding a SPECIFIC slot by ID is O(1).


  (A) unordered_map<string, ParkingSlot>  slots_
      --------------------------------------------
      KEY:   Slot ID string (e.g. "M-A1", "C-B2")
      VALUE: ParkingSlot (type, zone, Available or Occupied)

      WHAT IT IS FOR:
        The master list of ALL parking spaces in the car park.

      WHY NON-LINEAR MAP (not vector)?
        - Each slot has a UNIQUE ID — perfect for a labelled cupboard.
        - When a car leaves, we must free slot "C-B2" IMMEDIATELY by name.
        - With a vector we would search every slot: "is this C-B2? is this?"
        - With a map: slots_["C-B2"] or find("C-B2") -> O(1) jump.

      OPERATIONS:
        ADD slot       -> emplace("C-B2", slotData)    -> O(1)
        FIND by ID     -> find("C-B2")                 -> O(1)
        CHANGE status  -> slots_["C-B2"].status = ...  -> O(1)
        LIST all slots -> loop entire map              -> O(n)

      USED IN:
        Task 1 — configure parking slots
        Task 2 — mark slot Occupied on entry
        Task 4 — mark slot Available on exit
        Task 5 — reports showing slots


  (B) unordered_map<string, VehicleEntry>  activeVehicles_
      ----------------------------------------------------
      KEY:   Plate number (e.g. "RAB123A")
      VALUE: VehicleEntry (type, entry time, which slot)

      WHAT IT IS FOR:
        "Who is parked RIGHT NOW?" — only cars still inside, not history.

      WHY NON-LINEAR MAP?
        - Plate must be UNIQUE while car is inside (no double parking).
        - On EXIT we must find "RAB123A" instantly — O(1).
        - On ENTRY we check "is RAB123A already here?" — O(1).

      OPERATIONS:
        ADD on entry   -> emplace("RAB123A", entryData) -> O(1)
        CHECK duplicate-> find("RAB123A")                 -> O(1)
        FIND on exit   -> find("RAB123A")                 -> O(1)
        REMOVE on exit -> erase("RAB123A")                -> O(1)

      USED IN:
        Task 2 — register vehicle entry
        Task 4 — process exit (then erase from this map)
        Task 5 — show currently parked vehicles


  (C) unordered_map<VehicleType, int>  currentPrices_
      ------------------------------------------------
      KEY:   Vehicle type (Motorcycle, Car, or Truck)
      VALUE: Current price in RWF per hour

      WHAT IT IS FOR:
        The LIVE price board. Only 3 keys ever exist.

      WHY NON-LINEAR MAP?
        - At exit: "what is the Car price RIGHT NOW?" -> O(1) lookup.
        - On price update: change one key without touching history.

      OPERATIONS:
        READ price     -> currentPrices_[CAR]            -> O(1)
        UPDATE price   -> currentPrices_[CAR] = 1500      -> O(1)

      USED IN:
        Task 3 — fee calculation at exit, price updates
        Task 5 — show current tariffs


================================================================================
PART C — WHICH TASK USES WHICH STRUCTURE?
================================================================================

TASK 1 — Configure parking slots
  Structure:  slots_ (unordered_map)     NON-LINEAR
  Why:         Each slot has unique ID; fast add and lookup by ID.

TASK 2 — Vehicle entry
  Structures:  activeVehicles_ (map)     NON-LINEAR  — who is parked
                slots_ (map)              NON-LINEAR  — find & mark slot
  Why:         Fast duplicate check by plate; fast slot update by ID.
  Note:        Scanning for a FREE slot loops all slots -> O(n) — OK because
               a car park has a small number of spaces.

TASK 3 — Fees and prices
  Structures:  currentPrices_ (map)      NON-LINEAR  — live prices
                tariffPolicies_ (vector)  LINEAR      — load defaults once
  Why:         Instant price lookup when car exits.

TASK 4 — Vehicle exit
  Structures:  activeVehicles_ (map)     NON-LINEAR  — find & delete car
                slots_ (map)              NON-LINEAR  — free the slot
                transactionHistory_ (vector) LINEAR   — save receipt at end
  Why:         Fast find car and slot; append history like adding to a diary.

TASK 5 — Reports
  Structures:  slots_ (map)              NON-LINEAR  — list slots
                activeVehicles_ (map)     NON-LINEAR  — parked cars
                transactionHistory_ (vector) LINEAR   — history & revenue
  Why:         Walk through data and print. Vector scan is O(n) — fine.

PART E — WHY NOT USE ONLY VECTORS FOR EVERYTHING?
================================================================================

You COULD store everything in vectors, but it would be SLOWER for lookups:

  Problem: "Is plate RAB123A already parked?"
    Vector way:  check item 1, item 2, item 3... until found  -> O(n)
    Map way:     go straight to "RAB123A" door                 -> O(1)

  Problem: "Free slot C-B2 when car leaves"
    Vector way:  search every slot for "C-B2"                  -> O(n)
    Map way:     slots_["C-B2"].status = Available             -> O(1)

We USE vector WHERE:
  - We only ADD at the end (history receipts)
  - We often need to read EVERYTHING (revenue = sum all fees)
  - The list is a timeline of events in order

We USE unordered_map WHERE:
  - Each item has a UNIQUE name (slot ID, plate, vehicle type)
  - We need FAST find, update, or delete by that name

PART F — SAME ENTRY AND EXIT TIME (0 MINUTES) — SIMPLE EXPLANATION
================================================================================

Example:
  Entry: 10-06-2026 09:00
  Exit:  10-06-2026 09:00

Step 1 — How long was the car there?
  Same time minus same time = 0 minutes.
  Receipt shows: Duration = 0 minutes (the truth).

Step 2 — How many hours do we CHARGE?
  Assignment rule: partial hours round UP. Even 15 minutes = 1 hour.
  0 minutes is the smallest possible stay, so we charge MINIMUM 1 hour.
  Code says: if duration <= 0, billedHours = 1.

Step 3 — How much money?
  Fee = 1 hour x price per hour.
  Example: 1 x 1500 = 1500 RWF.

So the receipt shows BOTH:
  Duration:     0 minutes   (how long they really stayed)
  Billed Hours: 1           (what they pay for)
  Total Fee:    1500 RWF

This is allowed because exit is NOT before entry — it is equal.
Only exit BEFORE entry is rejected.
